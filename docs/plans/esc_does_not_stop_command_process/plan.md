## Plan

This replan supersedes the rejected implementation direction that made the Windows keyboard-enhancement path a blanket no-op. Base the bug fix on `docs/plans/esc_does_not_stop_command_process/rca.md`, keep `crates/tui/app/src/app/tests.rs::app::tests::windows_keyboard_enhancement_path_does_not_emit_csi_u_sequences` as the Esc/Ctrl+C repro input, and also treat `docs/plans/ctrl_shift_enter_submits_on_windows/rca.md` plus `docs/plans/ctrl_shift_enter_submits_on_windows/plan.md` as regression constraints.

The accepted fix must satisfy all four contracts at the same time:

1. Esc cancels an active workflow command action from the TUI, including while the composer says `No agent accepting prompts · draft retained · Esc cancels`.
2. Ctrl+C remains a global exit shortcut before composer edit/submit gating.
3. Shift+Enter and Ctrl+Enter continue to insert a newline on Windows terminals where the previous modified-Enter fix applied.
4. Legacy Windows console startup and restore remain safe; do not execute crossterm keyboard-enhancement commands through the unsupported Windows WinAPI command path.

Per the latest plan feedback, move terminal-mode setup into its own workspace crate instead of keeping it in `cowboy`'s `app.rs` or an app-local module. Add `crates/tui/terminal` with package name `cowboy-tui-terminal`. That crate owns terminal raw-mode lifecycle, alternate-screen/mouse/bracketed-paste/cursor setup, keyboard-enhancement setup/restore, and the platform keyboard strategy. The `cowboy` app crate should depend on it and keep TUI orchestration, runtime dispatch, drawing, app state, transcript rendering, and `input::handle_key_press`.

Existing terminal crates already cover the low-level terminal primitives: `crossterm` is already used for raw mode, alternate screen, bracketed paste, mouse capture, key events, and keyboard-enhancement commands; `ratatui` is already used for rendering; `tui-input` is already used for composer editing. There is no existing internal Cowboy crate that owns terminal lifecycle setup. The new `cowboy-tui-terminal` crate should be a product-specific seam over `crossterm` terminal-mode lifecycle and platform quirks, not a replacement for `crossterm`, `ratatui`, or `tui-input`. Keep `ratatui` rendering and `tui-input` composer state in the `cowboy` app crate.

Use platform file names `windows.rs` and `unix.rs`; do not create or keep a `non_windows.rs` module. Platform `#[cfg]` branches should live at the `cowboy-tui-terminal` module boundary, not scattered through `crates/tui/app/src/app.rs`.

The key design constraint is that Windows must not emit unhandled CSI-u keyboard-enhancement sequences. If the implementation keeps Windows keyboard enhancement disabled, it must prove Windows still delivers modified Enter through a terminal-level strategy, not just through input-handler unit tests. If the implementation re-enables CSI-u on supported Windows terminals, it must first add a Windows event normalization path that decodes the resulting CSI-u key sequences before they reach `handle_key_press`; raw CSI-u must not be sent to crossterm's Windows `WinAPI` event reader without an app-owned decoder.

Keep cancellation and keyboard policy out of workflow runtime, command action, and storage crates. Command actions already use `tokio::process::Command::kill_on_drop(true)`, so once Esc reaches `AppState::cancel_background_tasks`, aborting the workflow background task drops the command child and kills the process.

## Changes

1. Add a new workspace crate at `crates/tui/terminal` named `cowboy-tui-terminal`, add it to `[workspace].members`, `default-members`, and `[workspace.dependencies]`, and add it as a dependency of the `cowboy` app crate. The new crate should depend on `crossterm` for terminal control; do not reimplement raw mode, alternate screen, mouse capture, bracketed paste, or event primitives that `crossterm` already provides.
2. Move `TerminalModeGuard`, terminal screen enter/restore helpers, cursor-style setup, keyboard-enhancement setup/restore, and related terminal lifecycle tests out of `crates/tui/app/src/app.rs` into `cowboy-tui-terminal`.
3. In `cowboy-tui-terminal`, use platform modules named `windows.rs` and `unix.rs`. The Unix module owns the current non-Windows `KeyboardEnhancementFlags::DISAMBIGUATE_ESCAPE_CODES` push/pop behavior; the Windows module owns Windows legacy-console safety and any Windows keyboard strategy.
4. Rework the Windows keyboard setup in `crates/tui/terminal/src/windows.rs` around an explicit strategy instead of a boolean no-op/non-no-op split. The strategy must make it clear whether Windows is using plain crossterm WinAPI events or an enhanced CSI-u path with an app-owned decoder.
5. Keep the legacy-console guard: no Windows path may call `execute!`, `queue!`, or `ExecutableCommand::execute` for `PushKeyboardEnhancementFlags` or `PopKeyboardEnhancementFlags` because crossterm 0.29 returns `Unsupported` for that WinAPI command path.
6. Keep `crates/tui/app/src/app/tests.rs::app::tests::windows_keyboard_enhancement_path_does_not_emit_csi_u_sequences` unchanged as the named regression input. If terminal source moves, update only its fixture lookup so the same test still inspects the Windows terminal implementation and still fails on unhandled CSI-u emission.
7. Restore a terminal-setup-level modified-Enter regression. It must fail if Windows falls back to a no-op keyboard strategy without another documented and tested way for Shift+Enter/Ctrl+Enter to reach `handle_key_press` with modifiers.
8. If a supported Windows enhanced path is needed, add a small Windows keyboard sequence normalizer in the TUI terminal/input boundary before `input::handle_key_press`. It only needs to decode the keys that define this bug boundary: Esc, Ctrl+C, Shift+Enter, and Ctrl+Enter. Plain key handling should continue to use crossterm events.
9. Keep `crates/tui/app/src/app/input.rs` ordering unchanged or equivalent: Ctrl+C returns `KeyHandling::Exit` before composer gating, and Esc calls `state.cancel_background_tasks()` before composer gating.
10. Preserve the Unix keyboard-enhancement path, including `KeyboardEnhancementFlags::DISAMBIGUATE_ESCAPE_CODES`, unless the same app-owned decoder strategy replaces it with equivalent tests.
11. Remove or revise the rejected input-handler-only proof that claimed Windows modified Enter was preserved merely because `handle_key_press` handles already-modified Enter events. Input-handler coverage is necessary but not sufficient for the Windows terminal delivery bug.
12. Do not make the composer title conditional as the primary fix. `Esc cancels` should remain accurate because the terminal/input path delivers Esc, not because the UI hides the affordance.
