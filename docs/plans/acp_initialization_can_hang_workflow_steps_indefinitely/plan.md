# Plan

Implement the approved fix described in `docs/plans/acp_initialization_can_hang_workflow_steps_indefinitely/rca.md`. Bound the ACP `initialize` handshake with `AgentWatchdogOptions::recovery_operation_timeout_seconds`, explicitly dispose of a transport that cannot initialize, and return a contextual client error so the existing workflow retry and failure handling can run instead of leaving the step indefinitely `running`.

Use the investigator-added `crates/agent/acp/src/client.rs::connect_times_out_when_initialize_never_responds` test as the primary regression input. Retain that test and its paused-time reproduction; extend its assertions rather than rewriting or replacing it.

# Changes

- In `crates/agent/acp/src/client.rs`, introduce one bounded initialization path around the existing low-level `initialize()` handshake. The helper should use the configured recovery-operation timeout, preserve successful capability and agent-info parsing, and produce explicit context that identifies ACP initialization as the failed operation.
- When initialization times out or otherwise fails after a transport has been installed, take the transport out of the client and call `Transport::force_terminate()` within a second bounded window using the same recovery-operation timeout. Always return the primary initialization failure as the leading error. If force termination fails or times out, append that cleanup failure to the returned error and emit an error log; do not replace or suppress the initialization failure.
- Use the bounded initialization path for initial `connect_with_transport_and_options`, lazy transport reconnection in `ensure_transport`, and replacement initialization in `hard_recover_and_continue` so no transport activation can wait forever. Preserve the recovery-specific error context and existing session continuation behavior.
- Keep cleanup behavior deterministic: successful termination returns only the initialization error; a termination error returns the initialization error followed by `transport force termination failed: <cause>`; and a termination timeout returns the initialization error followed by `transport force termination timed out`. In all three cases, the transport must be removed from the client and dropped before control returns.
- Do not change prompt response, cancellation, retry-budget, workflow persistence, or progress-event semantics. The initialization error should continue through `ProductionAcpConnector` and `AcpClientFactory` as the existing recoverable `cowboy_workflow_agent::Error::Client` path.

3. Run `cargo test -p cowboy-agent-acp client::tests::initialize_timeout_preserves_force_termination_error -- --exact --nocapture && cargo test -p cowboy-agent-acp client::tests::initialize_timeout_preserves_force_termination_timeout -- --exact --nocapture`. Both tests must confirm one termination attempt and transport drop.
4. Run `cargo test -p cowboy-agent-acp client::tests::test_connect_and_initialize -- --exact --nocapture && cargo test -p cowboy-agent-acp client::tests::watchdog_hard_initialization_timeout_disposes_replacement -- --exact --nocapture`. 5. Run `cargo test -p cowboy-agent-acp`. All ACP unit and integration tests that do not require ignored live backends must pass.
