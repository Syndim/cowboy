# Plan

Keep the existing source-specific evidence pipeline, but correct the two contracts that still prevent reliable reproduction: evidence cardinality and prompt rendering. The current workflow permits multiple implementer records for one TODO while tester and reviewer prompts assume one record, and `examples/workflows/utils/context.lua` converts scalars with `tostring` and omits empty `comparisons` arrays. That makes one-to-one matching ambiguous and erases distinctions the semantic-equality contract requires.

Adopt exactly one evidence record per `(source, subject_kind, subject_id)`. That record contains the subject's complete ordered procedure, including every command and manual step. A subject can still execute multiple commands: each command record points to the single evidence record through `subject_kind`, `subject_id`, and its one-based `procedure_index`. Duplicate evidence records for the same source and subject are invalid and must block implementation completion, test passage, validation achievement, or review approval. Tester and reviewer records reproduce the same complete procedure one-for-one; validators likewise emit exactly one record per `VAL-NN` criterion.

Preserve the discriminated evidence schema and comparison rules already established. Implementer and validator records require `comparisons: []`; tester TODO records require exactly one implementer comparison; reviewer TODO records require implementer and tester comparisons; reviewer validation records require the matching validator comparison. Stable IDs and exact subject text remain the durable join keys.

Render evidence into downstream prompts with an explicit typed representation instead of `tostring`. Every scalar must identify its type and value so number `0`, string `"0"`, boolean `false`, and similarly shaped text remain distinguishable. Required structured fields must render their state even when malformed. In particular, `comparisons` must distinguish an empty array, a missing field, an invalid non-array value, and a nonempty array. For schema-known empty Lua tables, render `comparisons` as an empty array; for tables with non-array keys or gaps, render an invalid-array marker. Downstream agents then have enough information to reject malformed evidence without confusing it with valid empty comparisons.

Make review a global two-pass process. Pass 1 assesses every required TODO and validation criterion before any reviewer command or manual step runs. Emit exactly one `reviewer_assessments` record per subject, in plan TODO order followed by validation-guide criterion order:

```yaml
subject_kind: "todo"
subject_id: "TODO-01"
subject: "Exact task or validation-criterion text"
source: "reviewer"
proof_verdict: "sound" # sound | unsound
relevance: "Why this procedure proves the exact subject"
sufficiency: "Why it covers the material acceptance boundary"
safety_and_executability: "Why prerequisites and execution are safe and available"
currentness: "Why it matches the current plan, code, and evidence"
falsifiability: "Which plausible defect would make the procedure fail"
non_circularity: "Why the proof does not merely trust an oracle produced by the code under test"
submission_verdict: "valid" # valid | invalid
submission_issues: []
```

All six rationale fields are required nonempty strings, including for an `unsound` verdict, and must explain the subject-specific conclusion rather than repeat the field name. `submission_verdict` separately records whether the implementer/tester/validator records faithfully represent the assessed procedure; invalid submissions list concrete typed issues. Stable ID and exact subject text join assessments to plan/validation subjects and later reviewer evidence.

`submission_issues` is an ordered array of typed issue records:

```yaml
submission_issues:
  - source: "tester" # implementer | tester | validator
    code: "observed_result_mismatch"
    field: "observed_result"
    message: "Tester result does not match the declared procedure"
```

Each issue requires an affected upstream `source`, a stable `code`, an exact field/path, and a nonempty actionable message. The initial stable codes are `missing_record`, `duplicate_record`, `subject_mismatch`, `procedure_mismatch`, `expected_result_mismatch`, `observed_result_mismatch`, `invalid_comparison`, `stale_record`, `malformed_field`, and `unmapped_command`; extending this closed set requires updating the guidance and tests. `submission_verdict: valid` requires `submission_issues: []`. `submission_verdict: invalid` requires at least one issue. Issue source and field must identify the actual defective upstream value for that subject. Typed rendering must preserve issue scalar types and order, and issue messages must remain distinct from raw `user_feedback` and reviewer routing feedback.

For source/subject validity, TODO issues may name only `implementer` or `tester`, while validation-criterion issues must name `validator`.

Pass 2 is globally gated: no reviewer command or reproduction record may exist until Pass 1 has produced a complete assessment array for every required subject and every assessment has `proof_verdict: sound` plus `submission_verdict: valid`. If any proof is unsound, `replan_requested` wins, even when other submissions are invalid; return the complete assessment array and globally empty `reviewer_commands`/`reviewer_evidence`. If every proof is sound but any submission is invalid, return `changes_requested` with the complete assessments and globally empty reviewer command/evidence arrays. Only when all assessments pass may the reviewer reproduce every subject and return `approved` after all reruns match. The reviewer must not improvise substitute checks.

Reset every prior TODO to unchecked. The prior checked state had no nested per-item procedure, expected result, or implementation observation, so it did not meet the plan's own completion gate. This revised TODO section supplies the procedure and expected result for every stable subject. The implementer must add the observed result and matching single `implementation_evidence` record beneath each item before checking it; the tester must independently reproduce each checked item and return the corresponding tester record.

Keep this work in the starter workflow prompts, context renderer, existing Lua loader/runtime tests, and the already relevant frontmatter value-semantics test. No core, engine, store, or TUI change is needed.
